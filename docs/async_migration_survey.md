# Per-Subsystem Concurrency Survey
Eight subsystems were mapped in parallel. Each entry records how the subsystem is driven today, what blocks full-async, what is already correct asio, the shared-state / thread-safety assumptions, the conversion approach, and effort.

## main-loop + io_context (application core: manual frame loop, io_context lifecycle, frame pacing/idle-sleep, signal_set shutdown, ordered teardown)
**Effort:** large

**Current driving model**

A SINGLE io_context is constructed in main() (`boost::asio::io_context io_context;`, ~line 249) and is NEVER driven by io_context::run(). main() runs a hand-rolled cooperative frame loop — the `while (!shutdown)` loop in src/aqualink-automate.cpp (~lines 867-943). Per frame it: (1) drains ready Asio handlers with the NON-blocking `io_context.poll()` (:761, return value `handlers_run` counts handlers run); (2) advances the protocol stack via `protocol_task->Poll()` (:764); (3) sweeps per-device watchdogs via `Devices::Capabilities::Restartable::PollAll()` (:767); (4) calls `http_server->Poll()` (:775), `https_server->Poll()` (:781), `mqtt_integration->Poll()` (:785) to queue/kick fresh async work; (5) drains again with `io_context.poll()` (:788). Pacing: `frame_period` is 1000us real-time or `replay_frame_period` under capture replay (:742); when `replay_paced || !had_work` it busy-paces with `std::this_thread::sleep_until(frame_start + frame_period)` (:800-803). `had_work` is `protocol_task->Poll()` result OR `handlers_run > 0` (:793). A Tracy frame mark is emitted each iteration: `profiler.Get()->EmitFrameMark("MainLoop")` (:752 initial, :805 per-frame). The whole loop and every subsystem run on ONE thread — the no-lock concurrency invariant.

**Already async (keep as-is)**

A surprising amount is already correct asio and stays. The shutdown signal handler is a true async_wait: boost::asio::signal_set shutdown_signals(io_context, SIGINT, SIGTERM); shutdown_signals.async_wait([&shutdown](...) { shutdown = true; }) at src/aqualink-automate.cpp:745-750 — under run() this just calls io_context.stop() (or cancels the root co_spawn) instead of flipping a bool. All the lifecycle services already take io_context& and use asio::steady_timer + async_wait with a self-rearming pattern and ec/!m_Running guards: HistoryService (src/core/history/history_service.cpp:512-547 ScheduleFlush/Heartbeat/Purge), JandyStartupService (src/jandy/startup/jandy_startup_service.h:36 m_Timer + Tick), AlertMonitor (src/core/alerting/alert_monitor.h:110 m_CommsTimer + ScheduleCommsTimer), plus equipment_cache and scheduler services. These need no frame loop — run() already drives them; the only change is they no longer benefit from being co-located with a poll(). The WebhookSink is the canonical full-coroutine example already in the tree: boost::asio::co_spawn(m_IoContext, RunWithRetry(...), boost::asio::detached) at src/core/alerting/webhook_sink.cpp:199-202, with awaitable<void> AttemptHttp/AttemptHttps doing co_await async_resolve/async_connect/async_write/async_read (:49-104). Beast HTTP accept/read/write and the WS handshake/read are already async completion chains (src/core/http/server/http_server.cpp DoAccept/DoRead/DoWrite/DoWsAccept/DoWsRead). MqttClient's resolve/connect/TLS-handshake phase is already async. So the migration is concentrated in the frame loop + the four Poll()-pumped subsystems, not a green-field rewrite.

**Shared state & thread-safety**

The single-thread cooperative invariant is load-bearing and pervasive. The HubLocator DI container is explicitly unsynchronised — its own doc says 'The container is unsynchronised; it is constructed and consumed on the single cooperative poll() thread' (src/core/kernel/hub_locator.h:34) and it stores shared_ptr<void> in a plain std::unordered_map with no mutex (:78). DataHub / EquipmentHub / StatisticsHub / PreferencesHub are read and mutated with NO locks. boost::signals2 status/config signals (DataHub::ConfigUpdateSignal, EquipmentHub::EquipmentStatusChangeSignal, AlertTransitionSignal) are fired SYNCHRONOUSLY and INLINE during protocol decode (ProtocolTask -> ProtocolHandler_ReadOp_MessageHandler), and their slots run consumer code on the SAME stack: e.g. WebSocket_Equipment connects slots that read m_Connections and enqueue payloads (src/core/http/websocket_equipment.cpp:26-78), MqttHub connects slots that arm on-change publishes (src/core/mqtt/mqtt_hub.cpp:168,184), HistoryService writes SQLite from its slot (history_service.cpp:94). All of these touch state with zero synchronisation and assume the producer (serial decode) and consumers run on one thread. Restartable::s_Instances is a mutable process-global vector (restartable.cpp:14). Global singletons (ProfilerFactory, ProfilingUnitFactory, MessageGeneratorRegistry, Logging severity filter) are also accessed lock-free. CONSEQUENCE: a SINGLE-threaded io_context::run() preserves every one of these invariants for free (handlers still serialise on one thread) and is by far the lowest-risk target. A MULTI-threaded io_context would break all of the above and require either (a) wrapping every hub/signal consumer in ONE shared strand (effectively re-serialising and negating the threads), or (b) auditing and locking each hub — a very large, regression-prone change. Recommendation embedded in the analysis: go single-threaded run() first; treat multi-threading as a separate, later effort gated on making the hubs strand- or mutex-safe.

**Async conversion approach**

Target shape: replace the manual loop (aqualink-automate.cpp:754-806) with `boost::asio::co_spawn(io_context, AppMain(...), token)` for each long-running subsystem, then call `io_context.run()` on the main thread (single-threaded to preserve the no-lock invariant). Per subsystem: (1) ProtocolTask -> an `awaitable<void>` read loop that co_awaits async_read_some on the serial port, feeds the circular buffer + generator, and drains an outbound asio::experimental::channel<vector<uint8_t>> fed by EnqueueWrite/the write signals; remove the bool Poll() and the m_SingleReadPerPoll hack (replay pacing becomes a steady_timer in the replay producer). (2) MqttClient -> one client coroutine: keep the existing async connect chain, then in Connected state co_await async_read for inbound and co_await a publish channel for outbound, with steady_timers for keepalive ping, CONNACK timeout, reconnect backoff, and the periodic status/stats publishes currently in MqttHub::Poll — delete both Poll() methods. (3) HttpServer WS egress -> per-connection writer coroutine that co_awaits a channel/condvar signalled when Broadcast() enqueues, replacing HttpServer::Poll/TryWsWrite; accept/read stay as-is. (4) Restartable -> a single repeating steady_timer co_spawn'd once that calls CheckWatchdog across s_Instances (or per-instance timer rearmed in Kick), replacing PollAll. Startup: each subsystem is co_spawn'd where it is currently constructed (serial after :381, protocol after :426, http after :637, mqtt after :668). Shutdown: replace the `shutdown` bool with a cancellation_signal / asio::cancellation_slot threaded into each co_spawn (or simply io_context.stop() after signal_set fires), then preserve the EXISTING ordered teardown (aqualink-automate.cpp:812-896): cancel serial first (:816-820) to unblock the read coroutine, then reset protocol_task, stop alert_monitor/history/jandy/scheduler/cache/mqtt/http, Clear() routing, Unregister IRecordingController before serial_port.reset() (:888-889), Clear the generator registry, StopProfiling last. The signal_set, all timer services, and webhook_sink need essentially no change — they already cooperate with run().

**Key files:** `src/aqualink-automate.cpp`, `src/core/devices/capabilities/restartable.h`, `src/core/devices/capabilities/restartable.cpp`, `src/core/kernel/hub_locator.h`, `src/core/interfaces/ihub.h`, `src/core/protocol/protocol_thread.cpp`, `src/core/http/server/http_server.cpp`, `src/core/http/websocket_equipment.cpp`, `src/core/mqtt/mqtt_client.cpp`, `src/core/mqtt/mqtt_hub.cpp`, `src/core/mqtt/mqtt_integration.cpp`, `src/core/alerting/webhook_sink.cpp`, `src/core/alerting/alert_monitor.h`, `src/core/history/history_service.cpp`, `src/jandy/startup/jandy_startup_service.h`, `src/core/interfaces/iprofiler.h`

**Risks**

1) The no-lock single-thread invariant (HubLocator + all hubs + inline signals2 dispatch) is the dominant risk: ANY move to a multi-threaded run() silently introduces data races across DataHub/EquipmentHub/signals — must stay single-threaded run() unless the hubs are first made strand/lock-safe. 2) Ordering of the teardown sequence is subtle and correct today (serial cancel must precede protocol reset; the non-owning IRecordingController null-deleter handle MUST be Unregister'd before serial_port.reset(), aqualink-automate.cpp:882-889) — a co_spawn/cancellation rewrite must reproduce this exact order or risk a dangling alias or a read coroutine blocked on a live port. 3) The `had_work` pacing heuristic and ProtocolTask::Poll()'s bool return are load-bearing today; removing them is fine but anything that reads them (tests, MockReplayHarness which drives Poll() directly per test/unit/support/unit_test_mockreplayharness.h) must be migrated — the whole test harness assumes a synchronous, step-driven Poll() model and would need a run_one/co_spawn equivalent. 4) Capture-replay pacing currently lives in the frame loop (replay_frame_period, aqualink-automate.cpp:405-424,742,800) and in ProtocolTask m_SingleReadPerPoll — both must be relocated into a replay-producer coroutine or playback timing breaks. 5) ~~MQTT's Connected-state would_block state machine~~ **(RECONCILED 2026-06-30 — no longer applies: MQTT is now async-mqtt-backed; the library owns reconnect/keepalive/framing. The only residual MQTT concern is preserving the app-level drop-oldest publish queue `MAX_PUBLISH_QUEUE_SIZE` and the `CalculateReconnectDelay` backoff+jitter under any channel conversion.)** 6) WS egress backpressure (MAX_MESSAGE_QUEUE_SIZE drop-oldest in websocket_equipment.cpp:86) must survive the channel conversion. 7) Profiler frame marks (EmitFrameMark, aqualink-automate.cpp:752,805) lose their natural per-frame cadence under run() — frame boundaries must be re-defined (e.g. mark per serial chunk or per timer tick) or Tracy frame stats go away.

## serial + ProtocolTask (RS-485 read/write engine and protocol task)
**Effort:** large

**Current driving model**

Entirely poll-driven from the manual frame loop, NOT io_context::run(). main() calls `bool had_work = protocol_task->Poll();` once per frame at src/aqualink-automate.cpp:764, inside `while (!shutdown)` (754) that brackets the call with `io_context.poll()` at :761 and :788. ProtocolTask::Poll() (src/core/protocol/protocol_thread.cpp:184) runs DrainWrites -> DrainReads -> ProcessMessages -> DrainWrites synchronously and returns true if messages parsed or writes pending (:231), which feeds the loop's adaptive-sleep pacing (:800-803). Crucially the serial port itself is driven SYNCHRONOUSLY, not via asio: every layer calls blocking-style `read_some(buffer, ec)` / `write_some(buffer, ec)` made non-blocking by configuration, NOT async_read_some/async_write_some. ProtocolTask takes a `std::shared_ptr<Serial::SerialPort>` and `StatisticsHub` (protocol_thread.h:31-33) — it is given NO io_context/executor and spawns nothing. The executor only reaches the port IMPLs: `auto executor = io_context.get_executor();` (aqualink-automate.cpp:300) is passed to PhysicalSerialPortImpl/NetworkSerialPortImpl ctors (:347, :363), where it backs a `boost::asio::serial_port`/`tcp::socket` used purely for synchronous calls.

**Blocking / sync / poll-driven points**

| Location | What | Impact / target |
|---|---|---|
| `src/core/protocol/protocol_thread.cpp:107 (DrainReads) and :58 (DrainWrites)` | Synchronous m_SerialPort->read_some(buffer, ec) in an unbounded for(;;) loop until would_block/try_again/0 bytes; write_some in a while loop draining m_WriteQueue with manual m_WriteOffset partial-write tracking | This is poll-slice I/O, not async. The whole read-decode-dispatch-respond cycle is a synchronous slice per Poll(). Becomes one co_spawned coroutine: `for(;;){ n = co_await port.async_read_some(buf); ... }` with async_write replacing the manual offset loop. No more would_block sentinel handling — the coroutine simply suspends until bytes arrive. |

A slow handler (SLOW_MESSAGE_THRESHOLD_US=2000, :28) directly stalls the read slice. If the protocol coroutine runs on a strand this stays safe; if multiple io_context threads ever touch GenerateMessage it must be guarded. Registration is startup-only so it is fine. |

**Already async (keep as-is)**

Very little here is genuinely async — this subsystem is the most poll-bound in the codebase. What CAN stay: the underlying boost::asio::serial_port (physical_serial_port_impl.h:54) and tcp::socket (network_serial_port_impl.h:61) are real asio objects bound to io_context's executor (aqualink-automate.cpp:300), so they ALREADY support async_read_some/async_write_some/async_connect — only the SYNCHRONOUS overloads are currently called. No rewrite of the object model is needed, just the call sites. The TCP connect path uses asio resolver + asio::connect (network_serial_port_impl.cpp:95,105). The shutdown wiring is already asio: signal_set::async_wait (aqualink-automate.cpp:746) and serial_port->cancel() (:819) to unblock I/O. The MessageGeneratorRegistry consume-or-defer contract and the circular-buffer framing are execution-model-agnostic and carry over unchanged. The write path is already decoupled from producers via boost::signals2: ProtocolTask::ConnectWriteSignal<JandyMessage_Ack>/<IAQMessage_ControlDataResponse> (aqualink-automate.cpp:427-428, template at protocol_thread.h:48-64) Serializes into a buffer and EnqueueWrite()s — that producer/queue seam is exactly where an experimental::channel<std::vector<uint8_t>> would slot in.

**Shared state & thread-safety**

Single-threaded no-lock assumptions are pervasive and load-bearing. (1) MessageGeneratorRegistry::m_Generators — unsynchronised vector, documented as relying on the single poll() thread (message_generator_registry.h:58-63). (2) ProtocolTask internal state — m_WriteQueue (std::deque), m_SerialBuffer (boost::circular_buffer), m_ReadBuffer, m_WriteOffset (protocol_thread.h:75-82) — all touched without locks; the write SIGNAL handler (ConnectWriteSignal lambda) calls EnqueueWrite (protocol_thread.cpp:236) which push_backs onto m_WriteQueue from whatever thread fires the send-signal. Today every producer is on the poll thread so it is safe; a multi-threaded model needs m_WriteQueue to become a strand-posted or channel-fed structure. (3) The synchronous Signal_MessageWasReceived dispatch (imessagesignal_recv.h:46-68) fans every decoded message into DataHub/EquipmentHub/StatisticsHub and signals2 status signals with NO locks — the per-prompt single-thread invariant. (4) StatisticsHub counters (m_StatisticsHub->Serial.*, MessageErrors.*) are bumped as plain non-atomic increments throughout DrainReads/DrainWrites/ProcessMessages (e.g. protocol_thread.cpp:68,150,220). (5) boost::signals2 GetSignal()/GetPublisher() use function-local statics (imessagesignal_recv.h:39-43) — first-use init is thread-safe but concurrent connect/emit is not. A single-threaded io_context::run() preserves all these invariants for free; a multi-threaded io_context REQUIRES the protocol read-decode-dispatch coroutine to run on a dedicated strand so signal fan-out stays serialised, OR every hub/signal consumer must be made thread-safe.

**Async conversion approach**

Replace ProtocolTask::Poll() with a single co_spawned read-decode-dispatch coroutine bound to a serial strand: `awaitable<void> Run() { for(;;){ auto n = co_await m_SerialPort->async_read_some(asio::buffer(m_ReadBuffer)); insert into m_SerialBuffer; while(auto msg = GenerateMessage(buf)) co_dispatch; } }`. The unbounded synchronous read loop (DrainReads for(;;), protocol_thread.cpp:104) collapses to a single co_await async_read_some that naturally suspends instead of spinning on would_block — deleting the whole non-blocking-read apparatus (PhysicalSerialPortImpl::set_read_timeout(0) at physical_serial_port_impl.cpp:32,44 and the COMMTIMEOUTS/SO_RCVTIMEO platform shims, and m_Socket.non_blocking(true) at network_serial_port_impl.cpp:118). The single_read_per_poll replay bound (protocol_thread.cpp:166) becomes a co_await steady_timer between reads in the mock impl. For writes, feed an experimental::channel<std::vector<uint8_t>>: the ConnectWriteSignal handler co_spawns/posts a try_send onto the channel instead of EnqueueWrite onto m_WriteQueue; a separate writer coroutine `for(;;){ buf = co_await chan.async_receive(); co_await asio::async_write(port, asio::buffer(buf)); }` replaces DrainWrites and its manual m_WriteOffset partial-write bookkeeping (asio::async_write guarantees the whole buffer). Add a CancellationSlot so the SIGINT path (aqualink-automate.cpp:746) cancels both coroutines instead of serial_port->cancel() + protocol_task.reset() (:819,:823). co_spawn the pair onto io_context (or a per-port strand) right after the port opens (~aqualink-automate.cpp:426) and into a cancellation_signal stored for shutdown. ISerialPortImpl gains async_read_some/async_write_some (returning awaitable<size_t>) — the asio serial_port/socket already provide them; MockSerialPortImpl implements them over its file/replay with a timer; the RFC2217 IAC FilterInboundData (network_serial_port_impl.cpp:388-395) stays a post-read in-place transform. The MessageGeneratorRegistry and circular-buffer framing are reused verbatim. Decision point flagged for the cross-cutting plan: single-threaded io_context keeps all the no-lock hub fan-out safe (cheapest, recommended); multi-threaded REQUIRES the decode/dispatch coroutine pinned to a strand.

**Risks**

(1) The synchronous read-decode-dispatch chain means decode AND every downstream hub/device handler currently run on the read slice; co_spawning onto a multi-threaded io_context exposes ALL of DataHub/EquipmentHub/StatisticsHub + every signals2 slot to data races (no locks anywhere) — the read coroutine MUST share a strand with all consumers, or single-thread the io_context. (2) The write path crosses producer threads via signals2 (ConnectWriteSignal -> EnqueueWrite -> m_WriteQueue, protocol_thread.cpp:236): a naive channel.async_receive on a different executor than the signal emitter races m_WriteQueue; must post onto the writer's strand. (3) asio::async_write replaces the careful partial-write/would_block bookkeeping (m_WriteOffset, protocol_thread.cpp:55-93) — behaviour must stay equivalent for half-open TCP. (4) The RFC2217 IAC in-place filter mutates the read buffer AFTER read_some (network_serial_port_impl.cpp:388) and shrinks bytes_read — must run inside the async read completion before bytes enter the circular buffer, preserving order. (5) NetworkSerialPortImpl's self-healing disconnect (m_IsOpen flip + socket close on hard error, :373/:441) and the SO_RCVTIMEO/COMMTIMEOUTS shims are non-blocking-poll artifacts; deleting them while keeping reconnect semantics needs a deliberate async supervisor/reconnect coroutine. (6) MockSerialPortImpl::HandleMockWrite's std::this_thread::sleep_for (mock_serial_port_impl.cpp:326) would block an io_context thread under run(); replace with a timer. (7) Replay pacing currently lives in the frame loop (single_read_per_poll + replay_frame_period sleep) — moving it into the coroutine/mock without changing playback timing is subtle and is covered by the fixture-replay tests, which must keep passing.

**Async conversion approach**

Target shape under io_context::run():
1. Accept loop -> co_spawn(io, listener(), detached): a loop that co_awaits acceptor.async_accept(use_awaitable) and co_spawns a per-session coroutine. Connection-limit/erase bookkeeping becomes a simple counter on the acceptor strand instead of the m_Sessions sweep.
2. Per-session coroutine -> session_run() awaitable<void>: linearises the existing state machine — co_await ssl handshake, then a loop of co_await async_read / route / co_await async_write, mirroring DoRead/ProcessRequest/DoWrite. The chain of OnRead/OnWrite member callbacks collapses into straight-line code; self=shared_from_this lifetime is replaced by the coroutine frame owning the stream.
3. WebSocket: after upgrade, co_spawn TWO coroutines on a PER-CONNECTION strand — a reader (loop co_await ws.async_read) and a WRITER (the key change). The writer replaces today's poll-kicked TryWsWrite: it co_awaits a per-connection asio::experimental::channel<void(error_code,shared_ptr<const string>)>; Broadcast/OnOpen push into the channel instead of a raw deque, waking the writer immediately (eliminates the up-to-one-frame egress latency and the whole HttpServer::Poll()/HttpSessionState::Poll() path, http_server.cpp:54-62,680-712). The bounded channel naturally provides the MAX_MESSAGE_QUEUE_SIZE backpressure/drop policy (websocket_equipment.cpp:86-96).
4. Strands for shared state: bind all HTTP request handling + hub access to a SINGLE strand (preserving the lock-free hub invariant cheaply) OR make the hubs internally synchronised. Bind each WS connection's reader+writer+queue to its own strand; signal2 slots that feed WS queues must channel->async_send (or post to the connection strand) so the cross-thread handoff is safe.
5. Blocking handlers (history SQLite webroute_history.cpp:78-113, recording fs ops) -> co_await async work posted to a dedicated thread-pool/strand, so a slow query never stalls the io_context threads.
6. Lifecycle: drop http_server->Poll() (aqualink-automate.cpp:773-781) and the manual loop; co_spawn the listener at startup; on the existing signal_set shutdown (:745-750) fire a cancellation_signal / close the acceptor + channels so all session coroutines unwind (replacing HttpServer::Stop() :714-732 and Close() :69-91).

**Key files:** `src/core/http/server/http_server.h`, `src/core/http/server/http_server.cpp`, `src/core/http/server/server_types.h`, `src/core/http/server/routing/routing.cpp`, `src/core/http/server/routing/routing.h`, `src/core/interfaces/iwebsocket.h`, `src/core/interfaces/iwebroute.h`, `src/core/http/websocket_equipment.h`, `src/core/http/websocket_equipment.cpp`, `src/core/http/websocket_equipment_stats.cpp`, `src/core/http/websocket_event.cpp`, `src/core/http/server/responses/response_staticfile.cpp`, `src/core/http/webroute_equipment.cpp`, `src/core/http/webroute_equipment_button.cpp`, `src/core/http/webroute_history.cpp`, `src/core/http/webroute_diagnostics_recording.cpp`, `src/aqualink-automate.cpp`

**Risks**

- The lock-free hub invariant is the biggest sharp edge: ~30 webroute_*.cpp handlers read/mutate DataHub/Equipment/Statistics/PreferencesHub and ICommandDispatcher with no synchronisation (relying solely on single-thread). A naive multi-threaded io_context::run() introduces data races immediately. Safest path is single-strand HTTP (or all-on-one-io_context-thread) FIRST, multi-thread later only with hub locking.
- WS egress correctness: today exactly one writer per connection is guaranteed by the m_WsWriting flag + single thread (http_server.cpp:96,460,465,487). The channel-based per-connection writer must remain the SOLE writer; signal2 broadcasts must only enqueue, never write. Getting two coroutines writing the same ws stream is UB in beast.
- signals2 slots capture raw `this` and currently fire on the same thread that dequeues (websocket_equipment.h:46-50 uses scoped_connection to avoid UAF). Once WS queues live on per-connection strands, the slot (firing on the protocol thread) must hand off via channel/post; a synchronous slot touching strand-owned state would race.
- Cancellation/teardown ordering: the current Close()/Stop() force-cancels lowest layers (:75-90). The coroutine design must ensure cancellation_signals reach in-flight async_read/async_write and that channels are closed so writer coroutines exit — otherwise sessions leak on shutdown.
- WS handshake subprotocol echo + token-in-subprotocol logic (http_server.cpp:332-367, routing.cpp:148-178) and the per-branch security gating in HTTP_OnRequest (routing.cpp:340-385) are subtle; must be preserved verbatim when linearising into coroutines.
- TLS: SslStream async_shutdown semantics + expires_after handling (:517-541) are easy to get wrong in a coroutine rewrite (half-open / truncation).
- beast async_write takes ownership of the move-only message_generator for the op duration (:262-270) — the coroutine frame must keep it alive across the co_await.

## mqtt-integration (MqttClient / MqttHub / MqttIntegration / HomeAssistantDiscovery)
**Effort:** small — LARGELY OBVIATED (see reconciliation note)

> **RECONCILED 2026-06-30.** The original 2026-06-18 survey below described a
> HAND-ROLLED MQTT 3.1.1 client (a 6-state Poll() machine —
> Disconnected/Connecting/SendingConnect/WaitingConnack/Connected/Reconnecting —
> with `would_block` busy-polling, manual `m_WriteOffset` partial-writes, a
> hand-rolled `m_ConnectGeneration` staleness guard, and the claim that
> "vcpkg.json has no mqtt dependency"). **ALL of that is now false.** On
> 2026-06-19 the MQTT client was rewritten onto the **async-mqtt** library, so
> most of the migration work this subsystem was scoped for no longer exists. The
> material below has been replaced with the current reality; the original
> poll-machine description is retired.

**Current driving model**

MQTT is now LIBRARY-BACKED. `src/core/mqtt/mqtt_client.cpp` includes
`<async_mqtt/all.hpp>` and owns an `async_mqtt::endpoint` (a nested
`MqttClient::Impl` holds a TCP or TLS endpoint — `am::endpoint<am::role::client,
am::protocol::mqtt|mqtts>`), and `vcpkg.json` declares the dependency
`{ "name": "async-mqtt", "features": ["tls"] }`. The library owns all of the MQTT
3.1.1 wire concerns the old survey enumerated: framing/encode/decode, keepalive
PINGREQ cadence, CONNECT/CONNACK handshake, and the async read/write completions —
all driven by the host `io_context`.

The client is STILL pumped from the manual frame loop, but `MqttClient::Poll()`
(`MqttClient::Poll`, mqtt_client.cpp) is now a thin wrapper: when running it just
calls `m_Impl->FlushIfConnected()` to drain the app-level outbound publish queue
once connected. It does NOT itself do socket I/O — async_mqtt's own completion
handlers (handshake/recv/keepalive/reconnect) advance via the host
`io_context.poll()` inside the `while (!shutdown)` frame loop. `mqtt_integration->Poll()`
is still called once per frame (`mqtt_integration->Poll()`, ~line 916 in
src/aqualink-automate.cpp) and forwards down to `MqttClient::Poll()`. Hub-change
signals (DataHub ConfigUpdateSignal, EquipmentHub EquipmentStatusChangeSignal)
still set a debounce flag in MqttHub; the actual publish is enqueued and drained
by `FlushIfConnected()`.

**What actually remains to migrate (app-level glue only)**

The heavy MQTT machinery the original survey targeted is GONE — the library owns
it. Only thin app-level glue still rides the per-frame pump and would need to move
to a fully-async (`io_context::run()`) model:

| Symbol / location | What | Impact / target |
|---|---|---|
| `MqttClient::CalculateReconnectDelay()` (mqtt_client.cpp) | Reconnect backoff + jitter is computed by the app and applied via `m_ReconnectTimer.expires_after(...)` (an `async_mqtt`/asio `steady_timer`). | Already timer-based, not poll-based. Under `io_context::run()` this needs no change — it is already a `co_await`-equivalent steady_timer. |
| `MqttClient::Poll()` -> `Impl::FlushIfConnected()` (mqtt_client.cpp) | The one remaining per-frame call: drains the bounded outbound publish queue once connected. | Becomes a co_spawned writer that wakes on a channel/condition when `Publish()` enqueues, instead of being polled once per frame. | `MqttClient::MAX_PUBLISH_QUEUE_SIZE` (mqtt_client.h, value 1000) | Bounded drop-oldest publish queue: `Publish()` drops the oldest entry when the queue is at `MAX_PUBLISH_QUEUE_SIZE`. | Maps cleanly onto a bounded `experimental::channel` with drop-oldest-on-full, but is already correct as-is under the current model. |

**Already async (owned by async-mqtt — keep as-is)**

Connection establishment, the read loop, write completions, keepalive, and MQTT
framing/encode/decode are all owned by the **async-mqtt** library and run on the
host `io_context`. The TLS context setup (SNI + RFC6125 host-name verification) is
async_mqtt's `mqtts` predefined layer (`<async_mqtt/asio_bind/predefined_layer/mqtts.hpp>`)
plus Boost.Asio SSL. None of this is hand-rolled any more, so nothing here needs
porting — it is already coroutine/async-completion native under the library.

**Shared state & thread-safety**

MqttClient remains documented as NOT thread-safe — all public methods (Publish,
Poll, Start, Stop) must be called from the single io_context thread, and the
app-level state (`m_PublishQueue`, reconnect timer, published/dropped counters) is
accessed without synchronisation relying on single-threaded poll. The same
external writers cross into this state today and only work because everything is
one thread: (1) the AlertMonitor MQTT sink (set up in src/aqualink-automate.cpp,
the `mqtt_client->Publish()` wiring) calls Publish() from an AlertMonitor timer
handler; (2) the diagnostics HTTP route reads connection state / queue depth /
counters (webroute_diagnostics_mqtt.cpp) — safe ONLY because the handler runs on
the same io_context thread; (3) MqttHub publish methods read
DataHub/EquipmentHub/StatisticsHub via locked weak_ptr and the hub-change
signals2 slots fire on the protocol-decode hot path — currently the same thread.
A multi-threaded io_context would still break these no-lock assumptions and the
client would need to be bound to a strand.

**Async conversion approach**

Because async-mqtt already owns the connect/read/write/keepalive/framing, the
remaining work is small and app-level: (1) replace the single per-frame
`FlushIfConnected()` pump with a co_spawned writer that wakes when `Publish()`
enqueues (or feed a bounded `experimental::channel` directly, preserving the
`MAX_PUBLISH_QUEUE_SIZE` drop-oldest policy); (2) keep `CalculateReconnectDelay()`
+ the existing `steady_timer` reconnect as-is (already async); (3) delete the
per-frame `mqtt_integration->Poll()` call site and let `io_context::run()` drive
the endpoint; (4) if multi-threading is ever introduced, bind the client + the
external AlertMonitor/diagnostics entry points to the client's strand. There is no
longer a hand-rolled state machine, framing, partial-write bookkeeping, or
generation-token guard to re-express.

**Key files:** `src/core/mqtt/mqtt_client.h`, `src/core/mqtt/mqtt_client.cpp` (now async-mqtt-backed), `src/core/mqtt/mqtt_hub.h`, `src/core/mqtt/mqtt_hub.cpp`, `src/core/mqtt/mqtt_integration.h`, `src/core/mqtt/mqtt_integration.cpp`, `src/core/mqtt/ha_discovery.cpp`, `src/core/mqtt/mqtt_topic_scheme.h`, `src/core/http/webroute_diagnostics_mqtt.cpp`, `vcpkg.json` (declares async-mqtt + tls), `src/aqualink-automate.cpp`

**Risks**

1) Much LOWER than the original survey assumed — the library owns the wire
protocol, so there is no hand-rolled 3.1.1 state machine to faithfully reproduce.
The residual risk is confined to the app-level glue: the bounded drop-oldest queue
(`MAX_PUBLISH_QUEUE_SIZE`) drop policy and the reconnect backoff+jitter

2) Cross-thread external writers (the AlertMonitor sink and the diagnostics route)

must reconcile async_mqtt's endpoint teardown with any in-flight completions —
prefer the library's close/cancel rather than a synchronous tear-down.
4) Signal-slot threading: inbound-message handling still drives MqttHub command
handlers -> ICommandDispatcher, which mutate equipment/serial state on the
protocol side — those handlers must continue to run on the protocol strand/thread.

There are NO timers, channels, co_spawns, async_reads, threads, or mutexes anywhere under src/jandy/navigation or src/jandy/utility/screen_data_page* (grep-verified, zero hits). The Navigator, SpiderEngine, MenuModel and ScreenDataPage are deliberately pure, side-effect-light, synchronous state machines that take a screen snapshot + highlighted line and return an optional next key (std::optional<NavKeyCommand>). The genuinely-async part they hang off — the RS-485 read engine that decodes frames and fires the boost::signals2 slots which re-enter OnPageUpdate — lives OUTSIDE this subsystem (ProtocolTask / SlotConnectionManager). The clean separation (pure decision function vs. external driver) is exactly what makes the migration tractable: the state machine can keep returning a command while the DRIVER becomes a coroutine.

**Shared state & thread-safety**

Heavy reliance on the single-threaded no-lock invariant. (1) The MenuModel reference (Navigator::m_Model, navigator.h:158) is shared const graph data read on every OnPageUpdate via DetectPage/FindPath/GetPage/GetAllPages/GetIncomingSelectEdges — read-only, but FindPath does live BFS each call (spider_engine.cpp:279,291; navigator.cpp:344,902) with no synchronisation. (2) A single shared Navigator/keypad: OneTouchDevice owns exactly one m_Navigator + one m_SpiderEngine (onetouch_device.h:340-341); GoalInProgress() (onetouch_device.cpp:530-541) enforces that toggle/value-edit/boost/spa-switch/refresh goals NEVER interleave precisely because they all mutate the one Navigator with no lock. (3) Raw borrowed pointer m_pCurrentContent (navigator.h:191) is set to the stack screen snapshot at the top of OnPageUpdate and read by FindLineByLabel/AttemptWrapRecovery later in the SAME synchronous call — only valid because nothing can run between set and use on one thread; a suspension point (co_await) mid-OnPageUpdate would dangle it. (4) m_HighlightedLine, m_KeyCommand_ToSend, m_AckType_ToSend, m_OpState and every m_*StepCount / m_*InProgress goal flag on the device are plain members mutated from inside signal slots with no atomics/mutex. (5) The decrement/check ordering of m_PendingStatusMessages is explicitly order-critical: Slot_OneTouch_Status calls Navigator::OnStatusMessageReceived() BEFORE ProcessControllerUpdates(true) (onetouch_messageprocessors.cpp:113-124) and SpiderEngine::OnStatusReceived deliberately does NOT also decrement to avoid a double-decrement race (spider_engine.cpp:209-215) — a multi-threaded io_context would shatter this ordering contract. A multi-threaded run() would require a strand per OneTouch device (or per bus) to preserve all of the above.

**Async conversion approach**

Recommended shape: a single per-device strand carries one long-lived navigation coroutine; the pure Navigator/SpiderEngine/MenuModel stay as synchronous decision functions (do NOT rewrite the graph/BFS as coroutines). Introduce an awaitable page/status source: the Status/MessageLong slots push the decoded ScreenDataPage + highlighted line (and a Status tick) into an experimental::channel or a resumable awaitable, instead of synchronously re-entering OnPageUpdate. Then: (a) Replace the m_PendingStatusMessages counter + State::WaitingForPage with `co_await AwaitPage(expected_target, deadline)` and the cursor loop with `while(cursor!=target) co_await SendKeyAwaitStatus(dir)`; the frame-count limits (MAX_WAIT_CYCLES, MAX_BACK_PRESSES, MAX_CURSOR_MOVES, ONETOUCH_*_STEP_LIMIT) become asio::steady_timer-backed awaitable deadlines / cancellation_slots. (b) Make the SpiderEngine crawl a coroutine: `for each target: co_await NavigateTo(target); policy->OnPageReached(...)` — the State enum (Syncing/NavigatingToNext/CapturingPage) becomes ordinary control flow. (c) Turn the fire-and-forget goals into `awaitable<ActuationResult> ActuateDevice/SetPoolSetpoint/SetChlorinatorBoost/...` that co_spawn onto the device strand and co_return the Navigator's terminal IsSuccess()/IsComplete() result, so the web/MQTT command boundary can actually await the outcome instead of getting an immediate 'Accepted'. (d) Remove ProcessControllerUpdates' per-frame switch: there is no Poll() to remove (none exists), but the per-frame re-entrant dispatch is replaced by the coroutine awaiting page/status events. (e) Serialise via the device strand so the existing 'one shared Navigator, goals never interleave' invariant (GoalInProgress) is preserved without a mutex; the key-on-next-Status flush (m_KeyCommand_ToSend, onetouch_device.cpp:221-229) becomes `co_await SendKeyOnNextStatus(cmd)`. Startup: co_spawn the device's navigation coroutine when the device is constructed/comms-ready; shutdown: cancel it via the cancellation slot (replacing the watchdog/fault-state unwinding in WatchdogTimeoutOccurred/AttemptFaultRecovery).

**Risks**

(1) Re-entrancy + dangling borrows: OnPageUpdate today runs to completion synchronously and stashes m_pCurrentContent as a raw pointer to a stack ScreenDataPage; inserting a co_await mid-flow would dangle it. The page snapshot must be OWNED/moved into the coroutine frame, not borrowed. (2) The Status-counter ordering contract is subtle and load-bearing: OnStatusMessageReceived must run BEFORE the step function, and SpiderEngine deliberately does NOT double-decrement (spider_engine.cpp:209-215). Replacing it with a channel must preserve 'exactly one logical Status event per frame' or page transitions fire early/late. (3) 'Send key only on a Status ACK' (onetouch_device.cpp:221-229, V1_Normal vs V2_Normal at :129-133) is a hard protocol constraint, not an implementation detail — the awaitable that emits a key MUST resolve on a Status frame, not just any frame. (4) Multi-threaded io_context would break the no-lock single-Navigator/GoalInProgress invariant; a strand-per-device is mandatory, otherwise concurrent goals corrupt the shared Navigator. (5) Frame-count timeouts (MAX_WAIT_CYCLES=100, ONETOUCH_*_STEP_LIMIT=500, MAX_TRANSIENT_WAITS=20) currently scale with bus traffic rate, not wall-clock; converting them to real timers changes timeout behaviour and must be re-tuned against live captures. (6) A SECOND, separate scraping engine exists (Capabilities::Scrapeable, used by the PDA device) that DUPLICATES the same wait-stack/recovery/back-press handshake (scrapeable.cpp m_Stack_WaitingForMessage / m_Stack_WaitingForPage, MAX_RECOVERY_ATTEMPTS/MAX_BACK_PRESSES mirror Navigator's) — a full async migration should ideally converge both (deferred per the WU-NAV-SCREENDATA-GRAPH note in scrapeable.h:29-43) or they will drift. (7) Heavy existing test surface (test/unit/navigation/*, test/integration/flows/test_flow_onetouch_*) drives OnPageUpdate/ProcessStep directly and asserts on returned NavKeyCommand — those harnesses assume the synchronous return-a-command API and would need an awaitable-friendly equivalent.

**Async conversion approach**

The hubs have no Poll() to remove and should NOT be co_spawned — they are passive state. The migration question is purely *which concurrency discipline replaces the implicit single-thread invariant* once io_context::run() (possibly multi-threaded) drives the consumers. Three options, in increasing cost: (A) KEEP A SINGLE io_context THREAD — run() with one thread preserves the exact no-lock invariant; all signals2 fan-out stays inline and correct with ZERO hub changes. This is the lowest-risk target and should be the default; the manual Poll() loop is replaced by run() + each subsystem's work expressed as awaitables/timers, but the hub mutate-then-signal pattern is untouched. (B) STRANDS — if a multi-threaded io_context is required for throughput, bind ALL hub access to one shared strand (e.g. a `hub_strand` = make_strand(io_context)). Every writer (protocol processors) and every signal-driven consumer slot must `co_await dispatch(hub_strand, use_awaitable)` (or post) before touching hub state or firing a signal, serialising mutate+fan-out onto one logical executor while allowing I/O (serial read, beast accept/read, MQTT socket) to run on other threads. The signals2 scoped_connection lifetime model and the event-shared_ptr payloads (DataHub_ConfigEvent are already heap shared_ptr — safe to hand across the strand boundary) make this tractable. Consumers that do real I/O off the strand (WebSocket write, SQLite insert, MQTT publish) should take the event by shared_ptr on the strand, then hop to their own executor for the I/O. (C) EXPLICIT LOCKING — a shared_mutex per hub (read-shared for getters, unique for setters). Rejected as the primary approach: the DevicesGraph traversal + in-place AuxillaryTraits mutation and the synchronous signals2 fan-out would require holding the lock across slot execution (deadlock/reentrancy risk) or copying state out under lock (pervasive churn). Recommended path: target (A) first (single-thread run()), and ONLY if profiling demands parallelism adopt (B) strands, replacing the doc-comment invariants (equipment_hub.h:82-87, device_graph.h:161-166, websocket_equipment.h:52-56) with a hub_strand assertion. Converting the EquipmentHub/DataHub signals to an explicit `experimental::channel<...>` per consumer is an alternative to strands that decouples writer from reader thread entirely (writer posts the event, each consumer drains its channel on its own executor) and removes reentrancy, but multiplies the wiring vs strands.

The timer machinery across the whole subsystem is correct asio and stays: every service uses steady_timer + async_wait with an operation_aborted/!m_Running guard and re-arms itself (history_service.cpp:512-547, scheduler_service.cpp:300-316, equipment_cache_service.cpp:229-244, alert_monitor.cpp:161-183, jandy_startup_service.cpp:53-60). alert_monitor itself does zero blocking I/O — it is pure compute over hub state plus signals2 fan-out to sinks (alert_monitor.cpp:185-348). jandy startup is a pure state machine (Advance) with no I/O (jandy_startup_service.cpp:37-61). All of these map directly onto co_spawn + a steady_timer co_await loop.

**Shared state & thread-safety**

Every service was written for the single-thread no-lock invariant and would race under a multi-threaded io_context. (1) history: m_Buffer (vector), m_SeriesIds/m_SeriesLabels/m_LastSampleTs/m_DeviceMergeChecked (maps/sets) and the sqlite3* connection are mutated from BOTH the DataHub::ConfigUpdateSignal callback (OnConfigEvent) AND the flush/heartbeat/purge timer handlers AND the HTTP read API — all assumed same-thread (history_service.h:121-143). sqlite3 connections are not thread-safe across handles without serialization. (2) scheduler: m_Schedules/m_LastFiredMinute mutated by Tick (timer) and by Create/Replace/Remove (HTTP) with no lock (scheduler_service.h:87-88). (3) equipment_cache: reads/writes m_DataHub->Devices and PoolConfiguration/SystemBoard from the timer; ApplySnapshot calls m_DataHub->Devices.Add and ApplyPoolConfiguration (equipment_cache_service.cpp:93,101,168) — these touch the same DataHub the protocol thread mutates. (4) alert_monitor: subscribes to DataHub::ConfigUpdateSignal and EquipmentHub::EquipmentStatusChangeSignal (alert_monitor.cpp:125-133) and reads DataHub/StatisticsHub/PreferencesHub live; m_Latched mutated from both signal callbacks and the comms timer. (5) preferences: m_Hub fields (PreferencesHub) read LIVE by alert_monitor and history (e.g. AlertSaltLowPpm alert_monitor.cpp:198, HistoryRetentionDays history_service.cpp:414, AlertWebhookUrl via the webhook URL provider aqualink-automate.cpp:695) while preferences mutates them in ApplyJson — currently safe only because all on one thread. signals2 itself is emitted synchronously on the emitting thread (data_hub.cpp:19/27) so a multi-threaded io_context would deliver these callbacks on whatever thread emitted, breaking the no-lock assumption. A full-async multi-thread design needs each of these services pinned to a strand (or all sharing one logical-state strand) so hub reads/writes and signals2 delivery stay serialized.

**Async conversion approach**

Target shape: replace each ScheduleX()/async_wait self-rearm with a co_spawn'd awaitable loop: `co_spawn(strand, [this]()->awaitable<void>{ steady_timer t(co_await this_coro::executor); while(running){ t.expires_after(period); co_await t.async_wait(use_awaitable); co_await DoWork(); } }, bind_cancellation_slot(slot, detached))`. None of these has a Poll() of its own to remove (they are already async-timer-driven) — the only Poll() removal is the OUTER frame loop in main() that pumps them. Per service: (a) history — give it a dedicated DB strand (or a one-thread thread_pool executor) and make Flush/PurgeOld/EnsureSeries/ListSeries/QuerySeries awaitables that co_await onto that strand; OnConfigEvent should post the sample onto the strand rather than running EnsureSeries inline; the HTTP read API becomes awaitable<vector<...>> resolved on the DB strand, naturally serializing all sqlite3 access. (b) scheduler — co_spawn the 30s tick loop; move Load/Save onto a file-I/O worker strand and make Create/Replace/Remove co_await the save so the route awaits completion. (c) equipment_cache — co_spawn the 60s loop; Load/SaveNow on the file strand; Stop()'s final SaveNow becomes an awaited flush during graceful cancellation. (d) preferences — inject an executor/strand (it has none today), make Save/Load awaitable on the file strand, and have ApplyJson/RecordSpaSwitchAssignment be awaitables the routes co_await. (e) alert_monitor — co_spawn the comms loop; EvaluateAll stays sync but must run on the shared logical-state strand so signals2 callbacks and timer ticks are serialized. (f) jandy startup — co_spawn the 1s Advance loop; trivially cancellable. (g) webhook_sink — already correct; only swap asio::detached for a cancellation-slot-bound spawn so in-flight POSTs are cancelled at shutdown. Start()/Stop() map cleanly: Start() => co_spawn(strand, loop, bind_cancellation_slot(...)); Stop() => fire the cancellation_signal + (for history/equipment_cache/preferences) co_await a final flush. The existing Stop() already does cancel()+final-flush synchronously (history_service.cpp:105-122 flushes on stop, equipment_cache_service.cpp:222-227 SaveNow on stop), so the semantics carry over directly.

**Key files:** `src/core/history/history_service.h`, `src/core/history/history_service.cpp`, `src/core/history/sqlite_db.h`, `src/core/history/sqlite_db.cpp`, `src/core/scheduling/scheduler_service.h`, `src/core/scheduling/scheduler_service.cpp`, `src/core/equipment_cache/equipment_cache_service.h`, `src/core/equipment_cache/equipment_cache_service.cpp`, `src/core/alerting/alert_monitor.h`, `src/core/alerting/alert_monitor.cpp`, `src/core/alerting/webhook_sink.h`, `src/core/alerting/webhook_sink.cpp`, `src/core/preferences/preferences_service.h`, `src/core/preferences/preferences_service.cpp`, `src/jandy/startup/jandy_startup_service.h`, `src/jandy/startup/jandy_startup_service.cpp`, `src/jandy/startup/jandy_startup_environment.cpp`, `src/aqualink-automate.cpp`

## devices+dispatch (Jandy + Pentair device handlers, static message-signal dispatch, CommandDispatcher, spaside remote controller, Restartable watchdog)
**Effort:** large

**Current driving model**

Entirely Poll()-driven and slot-callback driven on the single cooperative thread; nothing in this subsystem owns an io_context or async chain of its own.

READ/DECODE drive: the manual frame loop calls protocol_task->Poll() -> ProtocolTask::ProcessMessages (src/core/protocol/protocol_thread.cpp:239-264) which, per parsed message, synchronously calls ProtocolHandler_ReadOp_MessageHandler (protocol_thread.cpp:264). That handler calls signal_base->Signal_MessageWasReceived() (src/core/protocol/protocol_handler_read_handler_messages.cpp:33), which fires the per-message static boost::signals2 signal (src/core/interfaces/imessagesignal_recv.h:46-68 GetSignal()/Signal_MessageWasReceived, using CRTP+static_cast on the hot path). Device slots registered via SlotConnectionManager::RegisterSlot<T> (src/core/utility/slot_connection_manager.h:49-58) and the Jandy dest-filtered RegisterSlot_FilterByDeviceId (src/jandy/utility/jandy_slot_connection_manager.h:29-38, src/jandy/utility/filtered_slot.h:30-49) run INLINE, synchronously, on the same call stack as the serial read. So every device handler (Slot_*) executes on the decode thread inside Poll().

WATCHDOG drive: Restartable::PollAll() (src/core/devices/capabilities/restartable.cpp:35-46) is called once per frame by the main loop; it reads steady_clock::now() once and walks a global std::vector<Restartable*> s_Instances (restartable.cpp:14) calling CheckWatchdog on each. This is NOT an asio timer -- it is a hand-rolled per-frame deadline sweep over a process-global registry.

CONTROLLER-UPDATE/WRITE drive: Jandy controllers act inside their own message slots. E.g. OneTouchDevice::ProcessControllerUpdates (src/jandy/devices/onetouch_device.cpp:127) runs the scrape/actuation state machine and emits the response ACK via Signal_AckMessage (onetouch_device.cpp:226/236) -- called from a Status-message slot, synchronously. SerialAdapterDevice::ProcessControllerUpdates (src/jandy/devices/serial_adapter_device.cpp:79-176) likewise dequeues one m_PendingCommands entry per master poll and emits it via Signal_AckMessage (serial_adapter_device.cpp:175). Outbound bytes go out by Signal_MessageToSend() (src/core/interfaces/imessagesignal_send.h:44-62) which fires the static send-publisher; ProtocolTask::ConnectWriteSignal's lambda (src/core/protocol/protocol_thread.h:48-64) serializes and calls EnqueueWrite, draining onto the wire via DrainWrites() (protocol_thread.cpp:210) -- all on the one thread.

**Already async (keep as-is)**

Almost nothing in THIS subsystem is asio-async today -- and that is largely fine because the work is event-driven off the serial decode, not time-driven.

- The outbound write actually reaching the wire IS already mediated by ProtocolTask's beast/asio-adjacent write queue (ProtocolTask::ConnectWriteSignal lambda + EnqueueWrite + DrainWrites, src/core/protocol/protocol_thread.h:48-64, .cpp:210/234-237). Devices never touch the port directly; they only raise a static send-signal. This decoupling is the right shape and would survive a co_spawn migration nearly unchanged.
- The static signals2 receive/send dispatch (imessagesignal_recv.h, imessagesignal_send.h) is a clean publish/subscribe seam: it already inverts control so devices don't poll for messages. In an async world the signal-firing site (protocol_handler_read_handler_messages.cpp:33) is the single natural place to co_spawn or post handler coroutines.
- WebRoute_Diagnostics_Matter (src/core/http/webroute_diagnostics_matter.cpp:50-110) is the ONE place in the device/diagnostics area that already uses asio::co_spawn + co_await http::async_read on a DEDICATED background thread and only exposes a cached snapshot to OnRequest -- a good template for "keep slow work off the main loop" but it is QR-code sidecar fetching, not core device dispatch.
- Restartable is protocol-agnostic and centralized (PollAll over one registry); converting it to asio timers is a localized change with a clear single owner.

**Shared state & thread-safety**

The entire subsystem assumes the single-thread, no-lock invariant and would break under a multi-threaded io_context:

- Static per-message signals: IMessageSignalRecv<T>::GetSignal() / IMessageSignalSend<T>::GetPublisher() return function-local static shared_ptr<signal> (imessagesignal_recv.h:39-43, imessagesignal_send.h:37-41). boost::signals2 connect/disconnect IS thread-safe by default, but the connections are made/torn down from device ctors/dtors (filtered_slot.h:48/54, slot_connection_manager.h:54) with NO coordination vs concurrent signal emission; and the slot bodies freely mutate device + hub state with no locks.
- Restartable::s_Instances is a process-GLOBAL std::vector<Restartable*> mutated by every device ctor/dtor (restartable.cpp:21/32) and walked by PollAll (restartable.cpp:42). Concurrent device construction (e.g. async device discovery) + a timer/PollAll sweep would be a data race today.
- CommandDispatcher reads EquipmentHub via ForEachDevice (command_dispatcher.h:78) and mutates DataHub traits in WriteThroughChlorinatorSetpoint (command_dispatcher.cpp:192) and SerialAdapter aux creation/ConfigUpdateSignal (serial_adapter/serial_adapter_commandprocessors.cpp:98-145) -- all unlocked, racing the decode thread which is also mutating those same devices/traits/signals.
- Per-device mutable state touched from a slot AND from a web/MQTT-originated command with no sync: OneTouch m_KeyCommand_ToSend (onetouch_device.cpp:221), SerialAdapter m_PendingCommands deque (serial_adapter_device.cpp:92/192), Spaside m_PendingButton/m_AwaitingButtonAck (spaside_remote_device.cpp:45/89). The web/MQTT side (webroute_equipment_button.cpp:154 ToggleByUuid; mqtt_integration.cpp:369/439/640) writes these via the dispatcher from a code region that is only safe because it runs on the same thread.
- Command results are returned SYNCHRONOUSLY and OPTIMISTICALLY (CommandResult::Success the moment a command is queued, before any wire transmission), and an optimistic cache write-through (command_dispatcher.cpp:166-171) papers over the lack of a real completion -- per MEMORY this optimistic 200-apply already masks a real aux-toggle no-op bug, so the missing await/confirmation has bitten before.

**Async conversion approach**

Recommended shape (single decode/device strand first, multi-thread later):

1. Pin all device-signal dispatch to ONE strand. At the fire site (protocol_handler_read_handler_messages.cpp:33) co_spawn/post the handler invocation onto a device strand so slot bodies keep their no-lock assumption while the decode read itself can become async. Slots stay synchronous-looking but run as the strand's continuation; no per-device locking needed if every slot + PollAll + CommandDispatcher run on that strand.

2. Make command dispatch awaitable with a real completion. Change Capabilities::DeviceActuator/SetpointController/ChlorinatorController to return awaitable<ActuationResult>; CommandDispatcher::ToggleByUuid etc. become awaitable<CommandResult> co_spawned from the HTTP/MQTT handlers. The completion should resolve when the device actually emits the ACK (OneTouch onetouch_device.cpp:226 / SerialAdapter serial_adapter_device.cpp:175), not when queued -- model the pending-command queue as an experimental::channel<PendingCommand> whose send completes (or a per-command cancellation_slot-aware promise) when ProcessControllerUpdates transmits it. This removes the optimistic-Success lie and the write-through hack (command_dispatcher.cpp:166-171) can become a post-confirmation update.

3. Replace Restartable::PollAll with per-device asio::steady_timer on the device strand. Kick() reschedules timer.expires_after; WatchdogTimeoutOccurred() becomes the timer's async_wait completion. Drop s_Instances + PollAll entirely. Each device owns its timer bound to the strand executor, so Kick/cancel are strand-serialized (no lock).

4. co_spawn at startup / cancel at shutdown. Devices are created in Jandy/Pentair Configure (equipment construction) and registered into hubs; the watchdog timers and any per-device command-drain loop get co_spawned with a cancellation_slot wired to the existing signal_set shutdown so device coroutines unwind deterministically before the io_context stops.

5. Spaside/Pentair command emission (SendButtonAck, SetSpeed/SetPower) post onto the same device strand rather than touching members directly, giving the same single-writer discipline as #2.

The static send-signal -> ProtocolTask write-queue seam (protocol_thread.h:48-64) can stay as-is; it is already an async-friendly decoupling point.

**Risks**

- The "synchronous, optimistic, return-Success-on-queue" command contract is load-bearing for callers (HTTP 200/MQTT result strings). Turning it into a real await-the-wire completion changes timing/semantics and risks new timeouts/hangs if the master never polls (no ACK ever emitted) -- needs a cancellation/timeout on the await. MEMORY already records an aux-toggle no-op masked by the optimistic 200; an honest completion could newly SURFACE that as a user-visible failure.
- boost::signals2 is internally thread-safe for emit, but the slot BODIES are not; naively running the io_context multi-threaded without pinning device dispatch to a strand instantly races every device member + the DataHub/EquipmentHub traits the slots mutate (e.g. serial_adapter_commandprocessors.cpp:98-145 creating aux devices + firing ConfigUpdateSignal).
- Restartable::s_Instances is a process-global mutated in ctors/dtors; if device construction ever moves off the strand (async discovery) the registry sweep races. Per-device timers fix this but every device class mixes in Restartable, so it is a wide blast radius (Jandy + all Pentair devices, restartable used in 7+ device types).
- The OneTouch/SerialAdapter "command only emits in response to a Status/master poll" rule (onetouch_device.cpp:221, serial_adapter_device.cpp:86) is a HARD protocol constraint -- an awaitable command cannot complete faster than the bus allows, so the coroutine must genuinely suspend across multiple frames; getting the channel/promise lifetime right (device may be destroyed / emulation suppressed mid-wait, emulated.cpp:19-23) is the sharp edge.
- Emulation suppression (Capabilities::Emulated, emulated.h:39-63) silently drops queued commands when a real device is detected; an awaitable command path must resolve those waiters (NotSupported) instead of hanging.
